# XRAG 项目系统开发情况与业务逻辑闭环总结

## 一、系统概览

**胸部X光智能辅助诊断系统**：面向医生/质控/管理员的病例管理、AI 报告生成、质控评估与危急值预警的一体化工作台。技术栈：后端 Spring Boot + MyBatis-Plus + openGauss/MinIO，前端 Vue3 + Element Plus，AI 通过外部 HTTP 服务（报告生成、CheXbert 评测、术语分析、润色）接入。

## 三、核心业务逻辑闭环

### 闭环 1：病例 → 影像 → 报告（主诊断流程）

```
[创建病例] → [上传影像] → (无报告时) 自动触发 [AI 报告生成]
                ↓
        保存到 MinIO + image_info
                ↓
        AsyncTaskService.triggerAutoGenerateReport(caseId, imageId, doctorId)
                ↓
        ReportService.generate(dto, doctorId)
                ↓
        RetrievalService.search(caseId, imageId, topK)  // RAG：取典型病例上下文
                ↓
        AiServiceClient.generateReport(图像, 相似病例文本)
                ↓
        写入 report_info，病例 report_status=AI_DRAFT
                ↓
        afterCommit → AsyncTaskService.triggerEvalAsync(reportId)
```

- **闭环点**：病例存在 → 有影像 → 无报告时上传即触发生成；生成后自动进入评测。
- **状态流转**：病例/报告：`NONE` → `AI_DRAFT` → `EDITING` → `SIGNED`（可重新生成仅限非 SIGNED）。

### 闭环 2：报告 → 评测 → 危急值预警

```
ReportService.generate 提交后
        ↓
triggerEvalAsync(reportId) → EvalService.evaluate(reportId)
        ↓
AiServiceClient.evaluateWithCheXbert(报告文本)
        ↓
写入 eval_result，report_info.quality_grade 更新
        ↓
createCriticalAlertsIfNeeded(report, aiResult)
  - 若标签为 Pneumothorax / Pleural Effusion 且置信度 ≥ 0.7 → 写 critical_alert（待处理）
        ↓
前端：待处理数角标、危急值列表、按病例查预警 → 医生响应（respond）
```

- **闭环点**：报告生成/再生成后自动评测；高危标签自动产生危急值；医生在处理列表中响应闭环。

### 闭环 3：RAG 检索（典型病例 → 报告生成上下文）

```
报告生成时：RetrievalService.search(caseId, imageId, topK)
        ↓
retrieveTypicalCases(excludeCaseId, topK)
  - 从 case_info 取 is_typical=1、非当前病例，按检查时间倒序取 topK
  - 每例取最新 report_info 的 findings/impression
        ↓
写入 retrieval_log；相似病例列表随 ReportGenerateDTO 传给 AI 生成
```

- **闭环点**：典型病例标记（病例 → 标记典型）后，成为后续新报告生成的参考上下文；当前实现为「典型病例池」检索，相似度固定 1.0，未用向量检索。

### 闭环 4：报告编辑与签发

```
医生在工作台：加载报告草稿 → 编辑 findings/impression
        ↓
ReportService.saveDraft(reportId, dto, editorId)
  - 更新 report_info 与 case_info.report_status=EDITING
  - 写入 report_edit_history
        ↓
医生确认 → ReportService.sign(reportId, doctorId)
  - report_status / case_info.report_status = SIGNED，记录 sign_time
```

- **闭环点**：从 AI 草稿到编辑历史再到签发，状态一致；签发后不可再编辑/重新生成。

### 闭环 5：术语规范（可选）

```
前端发起：TermService.analyzeReport(reportId, draftFindings, draftImpression)
        ↓
AiServiceClient.analyzeTerms(文本) → 返回建议列表
        ↓
写入 term_correction（同一 reportId 先删后插）
        ↓
前端展示建议 → 采纳 acceptCorrection(id) / 驳回 dismissCorrection(id)
```

- **闭环点**：报告内容 → 术语分析 → 建议持久化 → 采纳/驳回，与报告详情一起展示。

### 闭环 6：标注（影像层）

```
评测完成后：EvalServiceImpl.generateAnnotationsFromEval(report, eval, aiResult)
  - 按 CheXbert 标签映射到解剖区域，写入 image_annotation（来源=AI）
        ↓
医生可：手动矩形标注 → AnnotationService.create(dto)；删除；切换 AI/医生层显示
        ↓
从报告同步 AI 标注：POST /api/annotations/ai/{imageId}/{reportId}（用报告内容重新生成 AI 标注）
```

- **闭环点**：评测结果自动生成 AI 标注；医生可叠加人工标注；报告变更可重新同步 AI 标注。

### 闭环 7：病例批量导入

```
管理员：上传 CSV（检查号,患者匿名ID,性别,年龄,检查时间,检查部位,科室）
        ↓
CaseService.importCases(file) → 逐行解析，createCase(dto)
  - 重复检查号记入 ImportResult.errors
        ↓
返回 ImportResult（总行数、成功数、失败数、错误明细）
```

- **闭环点**：批量建病例 → 后续可单独上传影像并自动触发报告生成，与闭环 1 衔接。

### 闭环 8：操作审计

```
关键接口带 @OperationLog(type, detail)
        ↓
OperationLogAspect 环绕切面 → 调用成功后 OperationLogService.saveLog(实体)
        ↓
写入 sys_operation_log；管理员/质控在「操作日志」按条件分页查询
```

- **闭环点**：关键操作可追溯，与权限（医生/质控/管理员）配合形成审计闭环。

| 维度 | 状态说明 |
|------|-----------|

## 六、可优化或未闭环点

1. **RAG**：若需「语义相似」检索，可引入向量库与影像/报告 embedding，替代或补充当前典型病例池检索。
2. **报告生成触发**：当前仅「该病例尚无 report_info 时上传影像」触发；若业务需要「每张新影像都生成」或「多影像选主图」，需扩展触发策略。
3. **术语采纳**：acceptCorrection 当前仅更新 term_correction 状态；若需把采纳结果回写到报告正文，需增加对 report 的写回逻辑。
4. **前端**：拆分工台子组件、移除死代码与备份文件，可提升可维护性与一致性。

## 七、医生体验与减负评估（是否真的方便、能否减负 60%）

### 7.1 当前流程中医生实际在做的事（单例）

| 步骤 | 医生操作 | 系统行为 | 负担评估 |
|------|----------|----------|----------|
| 1 病例来源 | 手动「新建病例」填 7 项（检查号、患者ID、性别、年龄、部位、科室、检查时间）或等管理员 CSV 导入 | 无 HIS/ PACS 拉单、无扫码/号段自动建册 | **高**：建病例仍以手工为主，未省 |
| 2 影像 | 在工作台选病例后上传影像，或先上传再选病例 | 无报告时自动触发生成；有报告则不触发 | **中**：自动生成好；但需先「选病例」再上传，步骤可更顺 |
| 3 进入工作 | 在左侧列表点选病例 | 并发请求：病例详情、影像列表、报告、历史、相似病例、评测、术语、危急值 | **中**：选一次等一轮加载；无「待诊队列」连续处理 |
| 4 看 AI 草稿 | 报告 Tab 下看到只读的「影像所见/印象」 | AI_DRAFT 状态仅展示，不可直接改 | **中**：需再点「开始编辑」才进编辑，多一步 |
| 5 编辑 | 在两个大文本框内改 findings/impression；可选「AI 润色」「术语纠正」 | 术语纠正：弹窗勾选 → 确认替换；润色：整段替换 | **高**：大段编辑为主，无「按句改」「术语自动高亮+一键采纳」 |
| 6 签发 | 先「保存草稿」（或跳过），再点「签发」并确认 | 签发后不可撤销；必要时触发二次评测 | **中**：两步可合并；高置信度时无「快速签发」 |
| 7 下一例 | 再点左侧另一病例 | 重复 3～6 | **中**：无「下一条」「待审核队列」流，需自己找下一例 |

### 7.2 与「减负 60%、享受 AI 红利」的差距（结论：当前方案**不足以**达成）

- **入口未减负**：病例创建仍靠手工或批量导入，没有「今日待诊/待写报告」自动拉单，医生仍要「找病人、建册」。
- **动线不是「以 AI 为主」**：理想是「打开即见待审核 AI 报告 → 只改错/补漏 → 一键签发」。当前是「选病例 → 等加载 → 看报告 → 点开始编辑 → 大段改 → 保存 → 签发」，仍以「医生写报告、AI 辅助」为主，而非「AI 写、医生审」。
- **编辑成本高**：整段文本框编辑，术语需单独点「术语纠正」再在弹窗里选；无分句/分块编辑、无保存时自动术语检查与一键采纳，改一处要扫整段。
- **批量只做了一半**：有「批量生成」，但生成后仍要**逐例点进**编辑+签发，没有「批量审核/批量通过/批量签发」，对成批病例减负有限。
- **危急值未嵌入主流程**：在工作台有展示，但未强制「当前病例有危急值未处理则不可签发」或「今日危急值优先队列」，易被忽略或需切页面处理。
- **信任与可解释性不足**：RAG 相似病例未在前端突出展示（如「本报告参考了以下 3 例」）；高置信度时也没有「可快速签发」的明确入口，医生难以「放心少改」。

**综合判断**：当前系统在「AI 自动出报告 + 自动评测 + 危急值」上已形成闭环，能减轻**部分**书写与质控负担，但**距离「工作强度减少 60%、医生明显享受 AI 红利」仍有明显差距**，主要差在：入口自动化、动线以「审核」为中心、编辑轻量化、批量审核/签发、危急值嵌入主流程、以及可解释与快速签发。

## 八、升级设计方案：让系统脱颖而出

在保留现有闭环与数据模型的前提下，通过**动线重构 + 少量新增能力**，把系统从「AI 辅助写报告」升级为「以 AI 为首诊、医生做审核与把关」，真正向「减负 60%」靠近。

### 8.1 设计原则

- **医生主任务从「写」变为「审」**：默认动线是「看 AI 报告 → 改错/补漏/确认 → 签发」，而不是「打开空白再写」。
- **减少单例操作步数**：合并「开始编辑」与打开、合并「保存+签发」为「确认签发」，高置信度提供「快速签发」。
- **入口与队列优先**：优先展示「今日待诊/待审核」队列，支持「下一条」连续处理；病例来源尽量自动（HIS 拉单/登记台/号段）。
- **批量贯穿全流程**：不仅批量生成，还支持「批量审核」「批量签发」，列表即可完成通过/需改/签发。
- **危急值零遗漏**：当前病例若有待处理危急值则阻断签发或强提示；首页/工作台有「今日危急值」入口。
- **可解释与信任**：相似病例、置信度、漏诊风险在界面显性展示；高置信度时提供一键快速签发。

---

### 8.2 具体方案（可分阶段落地）

#### A. 入口与待诊队列（高优先级）

- **今日待诊/待审核队列**  
  - 默认首页或工作台顶部：按「检查时间」或「待审核优先」展示今日（或可选日期）病例列表。  
  - 支持「待写报告」「AI 草稿待审」「已编辑待签发」等 Tab 或筛选，医生一眼看到「接下来该处理谁」。
- **病例来源自动化（可选，与医院信息化程度相关）**  
  - 方案 1：**HIS 对接**——按科室/日期拉取待检列表，自动或半自动创建病例（只补影像上传）。  
  - 方案 2：**登记台批量入科**——登记台一次勾选多个检查号，系统批量建病例，医生端只看到「待上传影像」或「待审核」。  
  - 方案 3：**号段/条码**——扫描或输入号段，自动建病例并带出基础信息，减少手工 7 项填写。  
- **「下一条」工作流**  
  - 当前病例签发或标记「稍后处理」后，自动选中队列中「下一条」待审核病例，减少在列表中「找下一例」的认知负担。

**效果**：减少「找病人、建病例」时间，让医生把时间集中在「审报告、改错、签发」。

---

#### B. 以「审核」为中心的单例动线（高优先级）

- **打开即「可审」**  
  - 有 AI 草稿时：进入病例即默认「审核模式」——所见即所得，可直接在所见/印象区域**小块编辑**（如按句、按段），无需先点「开始编辑」再在大文本框里改。  
  - 若保留「开始编辑」：可改为「进入编辑并自动保存草稿」，避免「只读 → 编辑」的额外一步。
- **分句/分块编辑（中期）**  
  - 将 AI 生成的「影像所见/印象」按句或按段渲染为可点击块，点击某块即弹出小编辑器或行内编辑，改完即更新草稿，减少在大段文本中定位的成本。  
  - 术语建议：在分块或段落旁**内联展示**（如下划线+建议替换），支持「单条采纳」「本段采纳」「全部采纳」。
- **保存即检查术语（可选）**  
  - 保存草稿或点击「检查术语」时，自动调用术语分析，在正文中**高亮**待规范词，并提供「一键采纳全部建议」或按条采纳，无需单独弹窗勾选再替换（可保留弹窗为高级选项）。
- **「确认签发」一步完成**  
  - 当前：保存草稿 + 点签发 + 确认。  
  - 升级：提供**「确认签发」**按钮：若处于编辑态则先写草稿再签发，若已是 AI_DRAFT 且医生未改则直接签发；二次确认框可保留。  
  - **高置信度快速签发**：当 AI 置信度 ≥ 某阈值（如 85%）且医生未改内容时，显示「AI 报告质量较高，可快速签发」及**「快速签发」**按钮，点击后仅需一次确认即完成签发，减少心理负担。

**效果**：单例从「选病例→加载→开始编辑→大段改→保存→签发」收敛为「选病例→加载→改错/补漏（或直接快速签发）→确认签发」，步数减少、认知负担降低。

---

#### C. 批量审核与批量签发（高优先级）

- **列表侧批量操作**  
  - 在「待审核」列表中，每条病例显示：检查号、患者摘要、**AI 置信度、有无危急值**。  
  - 支持多选后：**批量「通过（待签发）」**、**「需修改」**（标记后进入待办，可写备注）。  
  - 对已「通过」或已编辑的，支持**批量签发**（二次确认：将批量签发 N 例）。
- **侧边/抽屉快速预览（可选）**  
  - 列表中点击某例，右侧抽屉或侧栏展示该例 AI 报告摘要 + 影像缩略图，不离开列表即可「通过/需改」，通过后再在列表勾选参与批量签发。

**效果**：大批量时不必逐例点进详情再保存再签发，列表内即可完成大部分审核与签发，显著缩短批量场景时间。

---

#### D. 危急值嵌入主流程（高优先级）

- **当前病例若有待处理危急值**  
  - 在报告区或签发栏明显提示：「本病例有 X 条危急值待处理」。  
  - **策略 1（推荐）**：未处理危急值则**禁止签发**，或签发前强制二次确认「已知晓危急值未处理仍签发」。  
  - **策略 2**：允许签发，但记录「签发时存在未处理危急值」供质控审计。
- **工作台/首页「今日危急值」**  
  - 固定入口（如顶部卡片或侧栏）：今日待处理危急值数量 + 列表，点击进入对应病例并定位到预警区域，处理后可一键返回队列「下一条」。

**效果**：危急值不依赖医生记得去另一页查看，而是嵌在主流程内，零遗漏、可追溯。

---

#### E. 可解释与信任（中优先级）

- **相似病例与 RAG 可见**  
  - 在报告编辑/审核区固定展示「本报告参考的相似病例」（当前已有 similarCases 数据）：展示 1～3 例的检查号、摘要（findings/impression 摘要），可展开看全文。  
  - 让医生感知「AI 不是黑箱」，增强对 RAG 结果的信任。
- **漏诊/过度诊断与置信度**  
  - 已有评测结果（漏诊风险、过度诊断）与置信度条；可在「快速签发」旁简短说明：「当前无漏诊风险提示，置信度 XX%」，便于医生决策。
- **（远期）向量 RAG + 相似度**  
  - 若引入向量检索，可展示「与当前病例最相似的典型病例及相似度」，进一步强化可解释性。

**效果**：医生更敢「少改多签」，尤其是高置信度 + 无危急值 + 有相似病例支撑时，真正享受 AI 红利。

---

#### F. 其他可选项（按需）

- **报告润色**：保持现有「AI 润色」入口，可放在分块编辑的段落旁或工具栏，避免大段重写。  
- **术语采纳回写**：采纳术语时除更新 term_correction 外，可选「同步回写至报告正文」，减少「采纳了但忘记粘贴」的问题。  
- **预生成与夜间任务**：若存在「当日新影像凌晨就到位」的流程，可增加定时任务：对当日新影像自动跑报告生成+评测，医生早上打开即为「待审核」队列，进一步把工作从「写」转为「审」。

---

### 8.3 实施顺序建议

| 阶段 | 内容 | 预期减负贡献 |
|------|------|----------------|
| **一期** | 待诊队列 + 「下一条」 + 确认签发（合并保存与签发）+ 危急值阻断/强提示 + 高置信度快速签发 | 单例步数↓、危急值零遗漏、高置信度少改 |
| **二期** | 列表批量审核 + 批量签发 + 今日危急值入口 | 批量场景时间大幅缩短 |
| **三期** | 分句/分块编辑 + 术语内联与一键采纳 + 相似病例与 RAG 前台展示 | 编辑负担↓、信任度↑ |
| **四期** | 病例来源自动化（HIS/登记台/号段）+ 预生成/夜间任务 | 入口时间↓、早上即审 |

---

### 8.4 小结

- **当前系统**：业务闭环完整、AI 已参与生成与评测，但动线仍以「医生写、AI 辅」为主，入口与编辑成本高，批量与危急值未贯穿，**尚不足以支撑「工作强度减少 60%」的体验目标**。  
- **升级方向**：通过**入口与队列、以审核为中心的单例动线、批量审核/签发、危急值嵌入、可解释与快速签发**，把主场景从「写报告」转为「审报告」，并在列表与队列层面做批量与危急值闭环，系统即可在「医生真正享受 AI、工作强度显著下降」上脱颖而出。  

以上方案可与现有 `业务逻辑闭环与开发情况总结` 一并作为产品与开发迭代的基准文档使用。

## 九、医疗场景安全与追责符合性

从**医疗合规、数据安全、责任可追溯**三方面评估当前系统是否满足「安全与追责」需求，并标出缺口与改进建议。

### 9.1 已具备的能力（符合性较好）

| 维度 | 现状 | 说明 |
|------|------|------|
| **身份与访问** | JWT 无状态认证；除登录/刷新/文档外均需登录；角色 DOCTOR/QC/ADMIN | 满足「谁在用系统」可识别；接口级未登录即 401。 |
| **角色权限** | 签发报告仅 DOCTOR/ADMIN；删除病例/用户/配置仅 ADMIN；删除影像/标注/预警响应 DOCTOR 或 ADMIN；操作日志仅 ADMIN/QC 可查 | 满足「谁能做敏感操作」的区分，利于最小权限与追责。 |
| **报告不可篡改** | 已签发报告（SIGNED）不可再编辑、不可重新生成；保存草稿/签发接口内校验状态并抛错 | 满足「签发即定稿」，防止事后篡改，符合医疗证据固定。 |
| **签发责任归属** | report_info 存 doctor_id、sign_time；签发时传入当前用户 ID | 满足「谁在何时签发」可查，支持事后追责。 |
| **修改可追溯** | report_edit_history 存 editor_id、edit_time、findings/impression 前后内容、edit_note | 满足「谁在何时改了哪些内容」，支持审计与纠纷回溯。 |
| **危急值响应可追溯** | critical_alert 存 responder_id、response_time、response_action、response_note | 满足「谁在何时如何响应危急值」，便于质控与追责。 |
| **操作审计** | @OperationLog 记录：操作类型、target_id、user_id、client_ip、api_path、elapsed_ms、created_at；失败也记并带错误信息 | 满足「谁在何时对何对象执行了何操作、从哪 IP、调了哪个接口」，支持安全与合规审计。 |
| **病例删除约束** | 有已签发报告的病例不可删除（CaseHasSignedReportException） | 避免误删已定稿病例，保证「有报告即有病例」可追溯。 |

综上：**身份、权限、报告不可篡改、签发/编辑/危急值响应与操作审计**已形成可追溯链条，在「谁、何时、对何对象、做了什么」上基本符合医疗场景的安全与追责需求。

---

### 9.2 缺口与风险（需补齐或加强）

| 维度 | 缺口/风险 | 建议 |
|------|-----------|------|
| **敏感操作权限未全覆盖** | 报告生成、保存草稿、报告润色等未加 @PreAuthorize，理论上任意已登录用户（含 QC）可调用 | 为「生成报告、保存草稿、润色」等接口增加 @PreAuthorize("hasAnyAuthority('DOCTOR','ADMIN')")，与签发权限一致；仅查看可保留给 QC。 |
| **报告/病例查看未审计** | 获取报告详情、病例详情、影像列表等「查看」类接口未打 @OperationLog，无法追溯「谁在何时查看了哪份报告/病例」 | 增加「查看报告」「查看病例」等只读审计（可异步、可采样），满足隐私与合规「访问可追溯」；注意日志中不落报告正文等敏感内容，仅记录 caseId/reportId。 |
| **影像删除与报告可追溯性** | 删除影像未校验「该影像是否已被某份已签发报告用作生成依据」；若允许删除，则报告与原始影像的对应关系可能被破坏 | 删除影像前校验：若存在以该 imageId 为 query_image_id 的 retrieval_log，且关联 report 已 SIGNED，则禁止删除或仅允许逻辑删除（保留归档）；或明确制度上「签发后影像不可删」。 |
| **操作日志留存与脱敏** | 日志表结构具备，但未明确：保留期限、是否归档、是否脱敏（如 detail 中是否可能带患者信息） | 制定日志保留策略（如 ≥6 个月或按院方规定）；确保 detail 不写报告内容/患者标识；必要时对历史日志做脱敏或访问控制。 |
| **批量操作审计粒度** | 批量导入病例、批量生成报告等若以「一次请求多例」实现，当前日志可能只记一条操作，target_id 可能只记一个或截断 | 批量操作在日志中记录「操作类型 + 数量 + 可选的首个/范围 ID」，或增加 batch_id 关联明细表，便于事后查「某次批量操作影响了哪些病例」。 |
| **会话与 Token 安全** | JWT 无状态、未看到服务端黑名单/失效策略（如登出后 token 立即失效、密码修改后旧 token 失效） | 若需「登出即失效」，可增加 token 黑名单或短期黑名单；敏感操作（如改密码）后使旧 refresh token 失效，并在文档中说明 token 有效期与续期策略。 |

---

### 9.3 与常见医疗合规要求的对照（简要）

- **责任可追溯**：谁签发、谁编辑、谁响应危急值、谁执行了敏感操作——当前**已支持**；建议补「谁查看了报告/病例」。
- **报告不可事后篡改**：已签发即不可编辑/不可重新生成——**已满足**。
- **操作可审计**：关键写操作带操作人、时间、IP、接口、目标——**已满足**；建议补只读审计与批量粒度。
- **数据最小权限与权限分离**：角色区分、敏感接口 @PreAuthorize——**大部分满足**；建议对报告生成/保存/润色补权限。
- **敏感数据保护**：日志不落报告正文、传输 HTTPS（部署层）、存储访问控制（DB/MinIO 权限）——当前日志未写正文；HTTPS 与库权限依赖部署与运维，需在环境中落实。

---

### 9.4 结论与优先改进

- **结论**：当前系统在**身份认证、角色权限、报告不可篡改、签发/编辑/危急值责任归属、关键操作审计**上已基本符合医疗场景的安全与追责需求；在「谁做的、何时做的、对什么做的」上具备可追溯能力。
- **仍不足以完全满足**的方面：部分敏感接口权限未收紧、**报告/病例查看无审计**、**已签发报告所依据影像的可删除性**、操作日志留存与脱敏策略、批量操作审计粒度、Token 失效策略等。
- **建议优先**：  
  1）为报告生成/保存草稿/润色补 @PreAuthorize；  
  2）增加「查看报告」「查看病例」的只读审计（不落敏感内容）；  
  3）删除影像前校验「是否被已签发报告引用」，若被引用则禁止删除或仅允许归档；  
  4）明确操作日志保留期限与 detail 字段脱敏/内容范围。  

完成上述补齐后，可更稳妥地声称系统在安全与追责维度**符合医疗场景需求**。

### 10.6 汇总与修复优先级建议

- **P0（合规/安全必改）**：已签发报告可撤回的合规策略；报告相关接口权限；影像删除与已签发报告关联校验。
