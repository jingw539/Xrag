1.总体架构设计
本系统采用分层单体架构设计，整体以一个 Spring Boot 应用为后端核心，前端采用 Vue 3构建 Web 界面，自下而上划分为环境层、数据层、逻辑层和表现层，各模块部署在同一服务器，通过清晰的分层和接口约束实现低耦合、高内聚，在保证开发效率的同时兼顾后期可维护与扩展。
1.1.设计原则
本系统遵循以下设计原则，保障系统的长期可维护性与扩展性。
（1）高内聚低耦合：前端通过 RESTful API 访问后端，前端仅负责页面展示和交互，后端负责业务逻辑与数据存储。后端内部按“表现层‑业务层‑数据访问层”分层，AI 推理服务通过统一接口封装，对上屏蔽具体模型实现细节，便于后续替换与升级。
（2）云原生：支持基于 Docker 容器进行一键部署，便于在教学机房、本地服务器或云主机之间迁移。
（3）安全优先：从设计阶段即考虑账号认证、角色权限控制、数据脱敏和传输加密，严格控制病例与影像数据的访问范围，确保敏感信息在存储和传输过程中均有相应保护措施。
1.2.应用架构
本系统采用分层解耦的单体架构，自下而上划分为环境层、数据层、逻辑层和表现层，并通过集成体系与医院现有信息系统（如 PACS、HIS、EMR）预留互联能力。
1.2.1.表现层
表现层面向最终用户，提供基于浏览器的放射科工作站和管理/质控视图，由 Vue 3 + Element Plus 实现。
（1）放射科工作站（核心管理系统）
面向放射科医生和科室主任的 PC 端 Web 工作台，是系统的核心入口，提供：
病例列表与检索：按时间、患者、检查号等条件查询胸部 X 光检查记录。
影像浏览：加载胸部 X 光影像，支持基本窗宽窗位调整与缩放浏览。
AI 辅助诊断：一键触发“相似病例检索 + RAG 报告生成”，生成可编辑的报告草稿。
报告编辑与签发：医生在 AI 草稿基础上修改、补充、确认并签发正式报告，符合“AI 只做辅助，人类医生负责最终诊断”的行业共识。
评测结果查看：查看 CheXbert 提取的 14 类病理标签及 F1 等指标，用于评估 AI 报告与参考报告的一致性。
（2）管理与质控视图
面向科室负责人或高级放射科医生，在同一 Web 前端中通过角色权限控制提供管理与质控一体化视图，无需单独开发新系统。主要功能包括：
用户与权限管理：维护用户账号、角色分配及基本操作权限。
系统参数配置：配置检索 TopK、相似度阈值、危急值标签、报告模板等全局参数。
报告质量与模型效果监控：查看报告质量等级分布、CheXbert 评分趋势、不同病理类别的性能表现等统计图表。
典型病例管理：整理和标记具有代表性的病例，用于教学展示和模型效果对比。
（3）开放接口（Open API）
后端提供标准 RESTful API，前端通过 Token 调用；同时在接口设计上预留：
从 PACS / 中间库获取影像路径或 DICOM 链接。
向 HIS/EMR 回写结构化报告结论和关键病理标签，方便临床医生在原有系统中查看结果。
1.2.2.逻辑层
逻辑层是系统的核心业务大脑，部署于同一 Spring Boot 应用内部，采用“3 个核心业务域 + 3 个公共支撑服务”的模块化设计。
三大核心业务域：
（1）病例与影像管理
定位：负责管理胸部 X 光检查相关的结构化信息与影像资源，是后续 AI 分析与报告生成的基础。
功能模块：
① 病例信息管理：维护患者匿名信息、检查号、检查时间、检查部位等。
② 影像信息管理：记录影像文件路径、投照体位、图像尺寸等元数据，为前端阅片与后端模型推理提供依据。
③ 报告生命周期管理：管理报告从“未生成 → AI 草稿 → 医生编辑 → 已签发”的全流程状态变更，记录修改历史与责任医生。
（2）智能分析与报告生成
定位：实现从影像到文本报告的端到端智能辅助诊断流程。
功能模块：
① 向量编码与检索：智能分析模块通过调用外部多模态模型服务，将胸片编码为特征向量，并基于向量检索获得 TopK 相似病例，为 RAG 提供证据基础；系统在架构上预留了采用 MedCLIP+FAISS 自建向量服务的扩展能力，后续可在具备 GPU 资源时平滑替换实现。
② RAG Prompt 构建：综合当前病例信息、相似病例摘要和临床约束（仅基于影像描述、不给治疗建议），在 AI 服务端或后端模块中构造结构化提示词。
③ 报告生成：调用多模态大模型接口生成胸片报告草稿，包含 Findings 和 Impression 等关键段落。
④ 人机协同编辑：将生成结果返回前端，支持医生修改与补充，最终形成可追责的正式报告。
（3）报告评测与质量控制
定位：对生成报告进行标签化和量化评测，为科研与科室质控提供数据支撑。
功能模块：
① CheXbert 标签提取：对 AI 报告和参考报告（如公开数据集标注报告）提取 14 类胸部病理标签。
② 一致性评估：计算标签层面的精确率、召回率和 F1 分数，用于衡量系统在不同病理上的表现。
③ 质控统计：按时间、病理类别、模型版本等维度统计评测结果，为模型迭代和科内质量改进提供依据。
三大公共支撑服务：
（1）配置与规则服务：
统一管理检索 TopK、相似度阈值、危急值标签、报告模板等系统参数，避免规则分散在代码中硬编码。支持在运行过程中在线调整配置，并通过数据库表（如 sys_config 等）持久化记录，便于审计与回溯。
（2）安全与权限服务：
实现基于 JWT 的用户认证，登录后发放访问令牌。采用 RBAC 模型进行角色权限控制，区分放射科医生、管理与质控人员、系统管理员等角色，在接口和菜单层面限制访问范围。结合操作日志，实现关键行为的审计和追踪。
（3）AI 调用编排服务：封装与外部或自建 AI 推理服务的调用逻辑，对上层业务提供统一的“相似病例检索”“报告生成”“CheXbert 评测”等接口。屏蔽模型类型、部署位置等底层细节，便于在资源允许的情况下，将外部 API 平滑替换为本地 MedCLIP + FAISS + 私有化多模态大模型等实现。
1.2.3.数据层
数据层为系统提供统一、可靠的数据存储能力，重点支持病例管理、报告生成与评测分析等核心功能：
（1）业务数据存储
采用 openGauss 关系型数据库存储核心结构化数据，包括用户信息、角色与权限、病例信息、影像元数据、AI 生成报告与最终签发报告文本、CheXbert 评分及统计结果等。
通过事务机制保证报告生成与签发过程中的数据一致性，为后续的数据分析和科研应用提供可信的基础数据源。
（2）分析数据与统计视图
将报告质量等级、模型评测结果和系统使用日志等数据按时间和模型版本进行汇总。
在 openGauss 中为质控看板构建聚合表或视图，简化前端统计查询逻辑，提高统计功能响应速度。
（3）文件与对象存储
使用 MinIO 对象存储保存胸片图像、示例 DICOM 文件和导出的 PDF 报告等非结构化数据。
在数据库中仅记录对象路径或访问 URL，将结构化数据与大体量文件解耦，简化备份和迁移流程。

1.2.4.环境层
环境层为系统运行提供必要的计算、存储和网络资源保障，在满足原型验证的前提下兼顾后续扩展。
计算环境：在一台服务器或云主机上部署 Spring Boot 后端和 Vue 前端静态资源，必要时部署独立的 AI 调用网关或轻量推理服务。
存储环境：使用本地磁盘或网络存储保存数据库数据、影像文件及向量索引文件，结合定期备份策略保证数据安全。
网络环境：通过 Nginx 等反向代理组件提供统一访问入口，前端仅通过 HTTP/HTTPS 调用后端接口，AI 服务和数据库端口仅对后端开放，必要时结合简单防火墙或内外网划分降低安全风险。
1.2.5.集成体系
软件集成：当前原型通过前后端分离架构自洽运行，不依赖外部信息系统即可完成“影像导入–智能分析–报告生成–结果展示”的闭环。
硬件集成：原型阶段主要通过本地文件上传和共享目录方式获取胸部 X 光图像，无需改造或直接控制真实影像设备；未来在医院部署时，可根据实际情况与现有 PACS 或影像工作站配置影像导出/共享目录映射，使系统能够自动从指定设备或影像服务器获取检查图像，同时沿用放射科现有阅片工作站作为前端访问终端，无需新增专用硬件。
接口预留：在数据模型与接口设计中预留检查号、检查时间、科室代码等字段，与医院 HIS/RIS/EMR 常用结构对齐，为后续对接 DICOM、中间库或 HTTP API 留出空间。
1.3.技术架构

本系统基于“Spring Boot + Vue 3 + openGauss + MinIO”的前后端分离技术体系构建，后端采用单体应用、容器化部署的架构模式。整体设计遵循高内聚、低耦合原则，通过标准化 RESTful 接口、统一认证与权限控制以及关系型数据库与对象存储的组合，支撑胸部 X 光病例管理、智能报告生成与报告评测等核心业务场景。
1.3.1.前端开发
前端主要面向放射科医生和管理/质控角色，统一采用 PC Web 形态，降低终端适配成本。
（1）技术栈与页面组织
技术选型：使用 Vue 3 作为前端框架，配合 Element Plus 组件库构建界面。
组件化设计：利用 Vue 3 组合式 API 提升组件复用性，将病例列表、影像预览、AI 报告编辑、质控看板等拆分为独立组件，在不同页面复用。
视图划分：在同一个前端项目中，通过路由和角色信息区分“放射科工作站视图”和“管理与质控视图”，实现界面隔离而无需多套前端代码。
（2）接口调用与状态管理
统一接口封装：使用 Axios 封装 HTTP 请求，在请求拦截器中自动注入 JWT Token，在响应拦截器中统一处理异常和登录失效。
状态管理：根据系统复杂度，采用轻量级状态管理（如 Pinia）缓存当前登录用户信息、角色权限和基础配置，减少重复请求。
（3）发布与部署方式
前端构建后生成静态资源文件，可直接由后端内置 Web 容器提供访问。
在需要统一入口或负载均衡时，可在后续部署中叠加 Nginx 等反向代理组件，实现路径转发与静态资源缓存。
1.3.2.业务发布
为降低部署复杂度，系统采用内嵌 Web 容器直接对外提供服务，不额外引入独立网关和反向代理。
Spring Boot 内置容器（Tomcat）：
①负责对外监听 HTTP 端口，承载所有 REST 接口与静态资源访问。
②通过配置静态资源目录，可以直接服务由 Vue 构建生成的前端静态文件，形成一体化部署包。
在未来需要水平扩展或接入院内统一入口时，可在外部叠加 Nginx 或其他反向代理组件实现统一入口与负载均衡。
1.3.3.程序开发
后端基于 Spring Boot 最新稳定版本构建，采用分层架构和 MyBatisPlus 数据访问框架，实现业务逻辑与持久层的解耦。
（1）开发框架与分层结构
Spring Boot：整合 Spring MVC、Spring Validation 等组件，提供自动配置和内嵌 Web 容器，简化 Web 应用搭建与部署。
分层架构：后端按照 Controller（表现层）– Service（业务层）– Mapper/Repository（数据访问层）进行划分，业务逻辑集中在 Service 层，数据访问细节封装在 Mapper 层，提升代码可读性与可维护性。
（2）持久层框架
MyBatisPlus：在 MyBatis 基础上提供代码生成、通用 CRUD、分页查询等增强能力，结合实体类与 Mapper 接口，实现对 openGauss 数据库的高效访问。
通过注解或 XML 映射配置复杂查询，既保持 SQL 灵活性，又减少大量样板代码。
（3）接口标准与文档
接口标准：采用 RESTful 风格设计后端 API，使用标准 HTTP 动词（GET/POST/PUT/DELETE）对病例、影像、报告、评测结果等资源进行操作，统一约定返回结构（状态码、提示信息、数据载荷）。
接口文档：集成 Swagger，在开发与测试环境中自动生成在线接口文档，支持接口列表浏览、参数说明与在线调试，提升前后端协同效率并方便后期维护。
1.3.4.公共服务
系统在后端封装了一系列通用技术组件，为各业务模块提供公共支撑能力。
（1）认证与权限（JWT + RBAC）
认证机制：用户登录后由后端签发 JWT，前端在后续请求中通过 Authorization 头携带 Token，后端在过滤器/拦截器中完成校验与解析。
权限模型：采用 RBAC（基于角色的访问控制）模型，区分放射科医生、管理与质控等角色，在接口和菜单层面基于角色进行访问控制，保证敏感操作（如报告签发、参数配置）仅对特定角色开放。
（2）日志与操作审计
记录用户登录、病例查看、AI 报告生成、报告修改与签发等关键行为，将操作用户、时间、目标对象、来源 IP 等信息写入统一日志表。为问题追踪、责任认定和质控分析提供可查询的审计数据基础。
（3）异常处理与统一返回
通过全局异常处理机制捕获系统内部异常，将其转换为统一格式的错误响应（错误码 + 提示信息），减少前端处理分支。避免在接口返回中暴露堆栈信息和数据库结构，提升系统安全性和可维护性。
1.3.5.数据存储
采用 openGauss 关系型数据库配合 MinIO 对象存储，实现结构化数据与影像文件的统一管理。
（1）关系型数据库：openGauss
存储内容：用户账号、角色与权限、病例基本信息、影像元数据（文件路径、拍摄时间、投照体位等）、AI 报告草稿与最终报告文本、CheXbert 评分及统计结果等核心结构化数据。
访问方式：通过 MyBatisPlus 结合 openGauss 驱动进行访问，在报告生成与签发等关键流程中使用事务机制，确保多表写入的一致性和可靠性。
（2）对象存储：MinIO
存储内容：胸部 X 光图像、示例 DICOM 文件、导出的 PDF 报告等非结构化文件。
集成方式：后端通过 MinIO SDK 将文件以对象形式存储，数据库仅保存对象路径或访问 URL，实现“结构化数据进数据库、文件进对象存储”的存算分离，便于数据备份、迁移与扩容。
（3）分析与统计数据
在 openGauss 中为质控与科研设计专门的统计表和视图，如不同模型版本的评测结果汇总、报告质量分布、系统使用日志汇总等。
前端可视化看板直接基于这些聚合结果查询，减少运行时复杂多表关联，提高统计查询性能。
1.3.6.AI 与外部服务集成
考虑到当前服务器仅提供 CPU 且资源有限，系统在技术架构上采用“统一接口 + 可替换实现”的方式集成外部 AI 服务。
（1）AI 调用编排服务
在后端封装独立的 AI 调用模块，对上提供“相似病例检索 + 报告生成”“报告评测”等统一接口，业务层无需关心具体模型或部署方式。
当前阶段优先调用外部多模态模型 API 完成图像理解与报告生成，系统负责传递影像地址、病例信息并接收生成结果。
在具备 GPU 资源后，可将该模块替换为自建 Python 服务，内部集成 MedCLIP 向量编码、FAISS 相似检索及本地/私有化多模态大模型推理，实现从外部服务到自建服务的平滑迁移。
（2）CheXbert 等评测模型
通过独立脚本或服务运行 CheXbert，对 AI 报告与参考报告进行标签抽取和指标计算。
评测结果以标签和评分形式回传后端，统一写入评测结果表，用于质控分析、模型效果对比和科研统计。
1.3.7.运行环境与部署方式
（1）操作系统与运行环境
系统部署于国产服务器操作系统 openEuler 上，利用其在安全加固和性能优化方面的特性，为应用运行提供稳定基础。
后端以 Spring Boot 可执行 JAR 形式运行，配合内嵌 Web 容器对外提供 HTTP 服务，无需额外安装独立应用服务器。
（2）容器化部署
容器引擎：采用 Docker 对应用及依赖组件进行容器化封装。
部署方式：
①将 Spring Boot 后端打包为 Docker 镜像；
②将前端构建后的静态文件打入后端镜像或以独立静态服务容器运行；
③使用 Docker Compose 管理 openGauss、MinIO 和应用容器的启动顺序、网络配置和数据卷挂载，实现一键部署、环境隔离与便捷的本地/服务器迁移。

1.4.数据架构
本系统采用“业务数据库 + 对象存储”的统一数据架构，以 openGauss 作为结构化数据的核心存储，以 MinIO 作为影像与报告文件的对象存储。系统通过标准化的数据模型和接口设计，既支持单机原型独立运行，又为未来与医院 HIS/EMR/PACS 等系统对接预留数据层扩展能力。
1.4.1.数据来源与接入
在当前原型阶段，系统主要通过本地录入与文件上传方式获取数据，同时在模型设计上兼容院内系统的字段，便于后续扩展。
（1）原型阶段数据来源
本地录入与导入：通过前端界面录入或批量导入患者匿名信息、检查号、检查时间等基础病例数据，用于模拟来自 HIS/RIS 的检查申请信息。
文件上传：通过 Web 界面上传胸部 X 光图像或 DICOM 文件，系统将其存入 MinIO，对应的文件路径和元数据写入 openGauss，模拟来自 PACS 的影像输入。
（2）院内系统预留接入方式
HIS/EMR：在病例数据表中预留检查号、就诊号、科室代码等字段，可通过中间库或 HTTP API 在后续版本中接入 HIS/EMR 的检查申请与诊断信息。
PACS：在影像元数据表中预留 DICOM 路径、Study/Series/Instance UID 等字段，未来可通过 DICOM、共享目录或院内影像网关实现与 PACS 的影像链接对接。
1.4.2.核心数据处理
系统围绕“病例与影像管理—智能分析与报告生成—报告评测与质控”三条主线组织和处理数据。
（1）基础数据处理
标准化存储：对录入或导入的病例信息、影像元数据进行格式校验（如日期、编号规则）、字段映射与编码统一（如检查部位、投照体位等），在 openGauss 中形成内部统一的数据模型，作为系统的“单一事实来源”。​
关联关系建立：通过检查号、患者匿名 ID、影像文件路径等关键字段，将病例记录、影像对象和报告记录进行关联，保证一次检查的多种数据可以被完整追踪。
（2）三条核心业务数据流
① 病例与影像管理数据流
输入：病例基本信息、影像上传记录、影像元数据。
处理：生成病例主记录、影像元数据记录，并与对象存储中的图像/DICOM 建立关联。
输出：为智能分析模块、报告生成模块和前端影像浏览提供统一的数据来源。
② 智能分析与报告生成数据流
输入：当前病例信息（结构化数据）、影像路径（MinIO 地址或本地路径）、相似病例向量索引。
处理：
通过 AI 调用编排服务向外部多模态模型或自建服务发送请求，生成相似病例列表和报告草稿。
将 AI 生成的报告文本及中间结果（如相似病例 ID 列表）写入 openGauss。
输出：为前端展示 AI 草稿报告、人机协同编辑和后续 CheXbert 评测提供数据基础。
③ 报告评测与质控数据流
输入：AI 生成报告和参考报告文本（如公开数据集标注报告）、CheXbert 模型输出标签。
处理：
通过 CheXbert 提取 14 类胸部病理标签，计算精确率、召回率、F1 等指标。
将标签和指标写入专门的评测结果表，并按模型版本、病例类型、时间等维度进行汇总统计。
输出：为质控看板、模型迭代效果对比和科研分析提供结构化指标数据。
1.4.3.数据清洗与服务
业务过程中产生的原始数据在入库与统计阶段经过必要的数据清洗与整理，转化为可直接用于分析和展示的高价值数据资产。
（1）数据清洗与完善
完整性校验：检查病例记录与影像文件、报告之间是否存在缺失关联，如缺少影像路径、报告状态不一致等，并通过简单工具或后台功能进行修复。
一致性修正：对报告状态、签发时间、评测结果等关键字段进行一致性检查，保证同一病例在不同表中的信息保持同步。
统计视图构建：在 openGauss 中创建面向分析的汇总表和视图（如按病理类别的 F1 得分、按时间的报告质量趋势），简化前端统计查询逻辑，提升查询性能。
（2）数据服务出口
质控与科研报表服务：向管理与质控视图提供报告质量分布、CheXbert 指标趋势、典型病例列表等统计数据，用于科室质控和学术研究。
智能分析反馈服务：将相似病例信息、评测结果等反馈给模型效果分析模块，为后续优化检索策略与 Prompt 设计提供依据。
接口数据服务（预留）：通过 RESTful API 暴露部分结构化报告数据和病理标签，为未来与 HIS/EMR 等系统对接提供数据出口。
1.4.4.数据应用场景
经过标准化和清洗后的数据，主要服务于以下三类对象：
（1）人员
放射科医生：通过病例列表、影像浏览、AI 报告与评测结果界面，快速获取结构化的病例信息和报告质量信息，辅助诊断与教学。
管理与质控人员：通过统计看板查看不同时间段、不同病理类别下的模型表现和报告质量，为科室管理、培训和质量改进提供依据。
（2）算法与服务
AI 报告生成服务：利用既有病例与报告数据作为示例和检索语料，持续丰富相似病例库和向量索引，提高检索与生成效果。
评测与迭代服务：利用积累的 CheXbert 标签与指标数据，对不同模型版本进行效果对比，支撑模型选型和迭代决策。
（3）外部系统（预留）
医院信息系统（HIS/EMR）：未来可通过接口将结构化影像报告与关键病理标签回写至院内系统，使临床医生在原有工作流程中直接获取本系统输出的报告结果。

2.2.1.系统入口与安全
功能定位：为系统提供统一登录入口和访问控制机制，确保胸部 X 光病例及报告数据在授权范围内安全使用。
（1）用户登录
登录方式：采用账号密码登录，用户通过输入登录名和密码进入系统。
登录失败提示：当账号不存在或密码错误时，系统给出明确但不泄露敏感信息的错误提示。
基本体验：支持“记住账号”（可选），方便医生在固定工作站上快速登录；长时间无操作自动退出，需要重新登录以防止他人误用。
（2）身份认证
JWT 令牌签发：登录成功后，后端为用户签发 JWT 令牌，令牌中包含用户 ID、角色信息及过期时间。
令牌校验与续期：前端在后续请求中通过请求头携带令牌，后端在拦截器中完成校验与解析；在需要时可提供令牌续期能力，避免频繁重新登录。
（3）访问控制
基于角色的菜单控制（RBAC）：系统预置“放射科医生”“管理与质控人员”“系统管理员”等角色，不同角色登录后加载各自菜单结构。
资源访问控制：后端在接口层根据用户角色校验访问权限，拒绝未授权访问并返回统一错误码；当用户未登录或令牌失效时，返回未授权状态并引导前端跳转至登录页面。
2.2.2.放射科工作站（PC Web）
2.2.2.1.病例与影像管理
（1）病例列表与检索
过滤功能：支持按检查时间范围、检查号、患者匿名 ID、性别、科室、报告状态（未生成、AI 草稿、编辑中、已签发）等条件组合筛选。
检索操作：输入关键字后点击【搜索】按钮，对检查号、患者匿名 ID 等字段进行模糊查询；点击【重置】清空所有筛选条件，恢复默认列表。
列表展示：显示序号、检查号、患者匿名 ID、性别、年龄、检查时间、检查部位、科室、报告状态、最近评测结果、签发医生、签发时间等字段，可按检查时间、报告状态、签发时间升降序排序。
行级操作：【查看】进入病例详情查看影像与报告内容；【删除】仅对管理员开放，用于清理测试或错误导入的病例记录。
（2）影像浏览
展示功能：在病例详情页展示胸部 X 光图像，显示投照体位、拍摄时间、图像尺寸等元数据。
交互功能：支持对图像进行缩放、平移等基础操作，方便医生查看局部细节；可选支持简单窗宽窗位调节以适配不同显示设备。
2.2.2.2.AI 报告生成
（1）相似病例检索
触发方式：医生在病例详情页面点击【生成 AI 报告】按钮，系统根据当前病例影像和基础信息触发检索流程。
功能说明：后端调用 AI 调用编排模块，向外部多模态模型 API 或预构建的向量索引发送请求，获取 TopK 相似病例；前端以列表形式展示相似病例缩略图、检查时间、主要病理标签、报告摘要等信息，支持点击查看详细报告。
（2）RAG 报告生成
生成流程：系统将当前病例信息、相似病例摘要及生成约束（如仅基于影像描述、不提供治疗建议）拼接为结构化 Prompt，提交给多模态大模型接口。
结果展示：大模型返回文本作为 AI 报告草稿写入数据库，并在前端展示；报告结构包括 Findings（影像所见）和 Impression（印象）等字段，对 AI 自动生成部分进行高亮或标记，便于医生区分。
异常处理：当模型服务异常或超时时，前端提示生成失败原因，并允许医生重新发起生成操作。
（3）报告编辑与签发
编辑功能：提供富文本编辑区域，支持对报告草稿进行修改、增删和重写，贴近日常报告书写习惯。
状态管理：
【保存草稿】：将当前内容保存为“草稿”状态，记录保存时间与操作者，支持多次迭代修改。
【签发报告】：在医生确认无误后，将报告状态更新为“已签发”，记录签发医生、签发时间等审计信息；已签发报告仅允许具有特定权限的用户撤销或修改，并保留历史版本。
审计与追踪：系统记录每次保存和签发的时间、操作者、客户端 IP 等信息，用于后期审计与责任追踪。
2.2.2.3.报告评测查看
（1）CheXbert 标签展示
功能说明：对 AI 报告和参考报告分别运行 CheXbert 模型，抽取 14 类胸部病理标签（如 Cardiomegaly、Edema 等），并在页面中进行对比展示。
展示形式：使用表格或矩阵形式显示每个病理的有/无状态，AI 报告与参考报告各占一列，对标签不一致的项目高亮标识，提醒医生重点核对。
（2）评测指标展示
功能说明：展示按病例或批次计算得到的精确率、召回率、F1 等评测指标，用于量化 AI 报告与参考报告的一致性。
展示形式：在病例详情页展示该病例对应的局部指标（如漏检病理、误报病理）；提供跳转入口到质控视图，查看按时间、模型版本、病理类别聚合的统计曲线和柱状图。
2.2.3.全管理与质控视图
功能定位：为科室负责人和质控人员提供用户与权限管理、系统参数配置、报告质量监控和典型病例管理等功能，支撑日常管理与科研活动。
2.2.3.1.用户与权限管理
（1）用户列表与检索
支持按用户名、姓名、所属科室、角色、账号状态（启用/禁用）等条件筛选；
支持输入用户名或姓名关键字后点击【搜索】进行模糊查询，点击【重置】恢复默认列表。
（2）用户操作
【新增】：创建新用户账号，设置登录名、初始密码、所属科室和角色。
【编辑】：修改用户基本信息、角色分配和账号状态。
【启用/禁用】：批量或单个切换账号状态，控制是否允许登录系统。
【重置密码】：为忘记密码的用户重置登录密码，首次登录后强制修改。
（3）角色与权限配置
支持配置“放射科医生”“管理与质控”“系统管理员”等角色，每个角色绑定一组菜单和按钮权限。

2.2.3.2.系统参数配置（含对接预留）
功能定位：集中管理影响 AI 行为、质控逻辑及后续系统集成的关键参数，支持在线调整而无需修改代码。
（1）检索与生成参数
配置相似病例检索数量（TopK）和相似度阈值；
维护 RAG Prompt 模板，包括报告格式要求、语言风格、是否包含特定字段等；
配置 AI 报告生成的超时时间与重试次数。
（2）评测与质控参数
配置 CheXbert 指标阈值，当某一病理 F1 低于阈值时在质控视图中标记为“待优化”；
维护危急值病理标签列表，对这些标签在前端进行高亮和提醒。
（3）对接与扩展配置（预留）
院内系统对接配置：预留 PACS 影像共享目录路径、中间库数据库连接信息、HIS/EMR 接口 URL 等字段，目前仅保存配置，未来在医院环境部署时可直接启用。
外部 AI 服务配置：预留多模态模型 API 地址、鉴权方式（Token/Key）、超时时间，以及 CheXbert 服务或脚本运行节点等信息，为后续替换或扩展模型提供统一入口。
2.2.3.3.报告质量与模型效果监控
功能定位：以数据看板形式，从宏观层面监控系统生成报告的质量和 AI 模型的整体表现，为质控和模型迭代提供依据。
（1）质量统计
按日/周/月统计报告总量、已签发数量、AI 草稿采用率、人工完全重写比例等指标。
支持按科室、医生、检查类型等维度过滤和对比，帮助识别工作负荷和质量差异。

展示不同病理类别下的 CheXbert F1、精确率、召回率曲线，支持按时间区间和模型版本切换。
提供列表显示模型表现较弱的病理类别，协助有针对性地优化模型或 Prompt 设计。
（3）数据导出
支持将统计结果导出为 CSV/Excel，用于论文撰写、科研分析或管理汇报。
2.2.3.4.典型病例管理
功能定位：沉淀具有代表性和教学价值的病例，为教学、科研和模型评估提供样本库。
（1）典型病例标记
在病例详情页，具有相应权限的用户可点击【标记为典型病例】或【取消典型】对病例进行标注。
标记时可填写标签（如“经典肺炎”“生成效果优秀”“易混淆病例”等）和备注说明。
（2）典型病例列表
提供专门的典型病例列表页面，支持按病理类别、标记人、标记时间、评测指标等条件筛选。
列表项可直接跳转至病例详情，用于教学讨论、科内分享或模型效果展示。
2.2.3.5.系统管理（日志与审计、数据导入与初始化）
功能定位：为具有管理权限的用户提供日志审计和数据初始化能力，方便日常维护和原型演示。
（1）日志与审计查看
访问日志：记录每次请求的时间、接口路径、用户、来源 IP、执行耗时等信息，支持按时间范围、用户、接口路径等条件筛选与搜索。
操作审计：记录登录、退出、报告生成、报告签发、参数修改、用户管理等关键业务操作，支持查看指定用户在某一时间段内的操作轨迹，便于问题排查和安全审计。

数据导入：支持上传符合模板格式的病例信息文件（如 CSV/Excel），批量导入病例基础数据；支持批量上传影像文件和参考报告文件，并通过文件名或映射表与病例记录关联。
数据初始化：提供“一键清理测试数据”功能，仅限管理员使用，用于多次演示或重新实验时快速恢复系统初始状态。

4.原型交互设计
4.1.交互设计原则
一致性：全系统的按钮样式、弹窗风格、反馈提示语保持统一，降低用户学习成本。
即时反馈：任何用户操作（点击、提交、加载等）必须在 200ms 内给予视觉反馈，如 Loading 态、Toast 提示。
防错与容错：关键操作（如删除、支付）需二次确认；表单输入提供实时校验（如手机号格式错误立即标红）；网络异常时提供友好的“重试”引导，而非直接报错代码。
简洁高效：核心路径步骤最少化，去除冗余信息干扰。


4.2.关键页面原型
4.2.1.病例诊断工作台
功能定位： 放射科医生进行胸部X光阅片与报告撰写的核心工作台。集中展示病例基本信息、AI自动分析结果、原始影像与病灶标注、AI生成报告草稿及相似病例检索结果，并提供人工干预（编辑/签发/退回）及全流程节点追踪功能。
用户角色： 放射科医生。
前置条件： 用户已登录且拥有"报告编辑"与"报告签发"权限；存在状态为"待生成"、"AI草稿"或"编辑中"的病例记录；对应病例已完成影像上传并通过AI分析流程。
页面布局：
采用"左列表 - 右详情"分栏布局，兼顾批量病例处理效率与单条病例信息的深度查看。
左侧病例列表区占据页面左侧约 30% 宽度，支持按报告状态（待生成 / AI草稿 / 已签发）筛选过滤，并对存在AI预警的病例以橙色圆点高亮提示，供医生快速定位优先处理对象。
右侧病例详情区占据页面右侧约 70% 宽度，是核心作业区域，自上而下分为五个板块：
（1）操作按钮栏，配置当前病例所需的全部功能入口，包括重新触发AI报告生成、将病例标记为典型病例、查阅患者历史病历参考、查看处方全程追溯记录及打印输出。
（2）病例信息（对应处方笺），以 8 格元数据网格形式，展示病例的患者匿名ID、性别年龄、检查科室、检查时间、检查部位、投照体位、是否典型病例、当前报告状态等关键字段；下方附主诉与临床备注标签行；底部呈现 AI 初审结果条，以橙色预警色调展示 CheXbert 模型检测到的异常类别、风险说明及置信度进度条。
（3）影像查看与报告编辑区（对应药品明细列表），采用左右分栏结构：左侧为影像查看器，展示胸片原始影像及 AI 自动标注的病灶位置（红色脉冲圆环 + 置信度标签），并提供放大、缩小、旋转、窗宽窗位调节及手动标注工具；右侧为报告编辑面板，呈现 AI 生成的"影像所见"与"影像印象"文本，医生可直接在此修改，面板底部显示当前模型版本与整体置信度。此外，该区域下方附相似病例检索模块，以缩略图卡片形式展示 Top-3 向量相似度匹配结果，辅助医生对照参考历史典型病例。
（4）处理进度，以横向五节点步骤条形式，可视化展示当前病例所处的流程节点：影像上传 → AI 分析 → 医生编辑 → 签发报告 → CheXbert 评测。已完成节点显示绿色对勾，当前活跃节点以蓝色脉冲动效突出，待处理节点以灰色待机样式呈现，支持全程溯源。
（5）底部操作栏，始终固定在详情区底部，左侧提供"上一份 / 下一份"快捷翻页及当前 AI 预警状态提示，右侧提供三个核心操作按钮：保存草稿（随时保存编辑中的报告内容）、退回重拍（触发带原因选择的二次确认弹窗，通知技师重新拍摄）、签发报告（触发带备注填写的二次确认弹窗，签发后报告状态变更为 SIGNED 并进入 CheXbert 评测流程）。


4.2.2.报告管理列表
功能定位： 对系统内所有病例报告进行统一检索、状态监控与批量管理的列表工作台。支持按检查时间、报告状态、申请科室等多维条件筛选，并提供数据导出功能。
用户角色： 放射科医生、质控管理员。
前置条件： 用户已登录；系统中存在至少一条病例报告记录。
页面布局：
页面整体采用垂直纵向布局，自上而下分为两个区域：
顶部为多条件筛选栏，提供检查时间范围选择、报告状态下拉筛选（NONE / AI_DRAFT / EDITING / SIGNED）、申请科室下拉筛选及关键词搜索输入框，右侧配置查询、重置与导出操作按钮，支持快速定位目标报告。
下方为数据列表区，以表格形式呈现所有报告记录，列字段包括病例号、患者信息、检查时间、申请科室、报告状态（以彩色徽标区分）、AI 置信度（以数值及颜色分级展示）、签发医生及 F1 评分；每行提供"查看"操作按钮，直接跳转至病例诊断工作台对应病例的详情页。列表底部配置分页控件，支持按需调整每页条数。

4.2.3.评测统计
功能定位： 以数据看板形式，对 AI 报告生成模型的整体性能指标进行可视化呈现，辅助质控管理员与系统管理员监控模型质量、追踪优化趋势及进行版本横向对比。
用户角色： 质控管理员、系统管理员。
前置条件： 用户已登录且拥有"评测统计"查看权限；系统中存在已完成 CheXbert 评测的签发报告记录。
页面布局：
页面采用垂直纵向卡片式布局，自上而下分为三个层次：
顶部为 KPI 核心指标行，以四格卡片形式展示本月报告总量、宏平均 F1 均值、AI 报告医生采纳率、平均报告生成时长等核心数值，每项指标附环比变动趋势（上升绿色 / 下降红色），快速反映系统当前整体运行状态。
中部为双栏图表区：左侧展示近 6 个月宏平均 F1 趋势的纵向柱状图，直观呈现模型性能的月度演变走势；右侧展示14类病理标签（肺不张、心脏肥大、胸腔积液、肺结节、气胸、肺实变等）各自 F1 分值的横向条形图，以颜色分级（蓝色优秀/橙色中等/红色较弱）标注各类别的当前检测能力短板。
底部为模型版本对比表，以结构化表格形式横向对比当前版本与历史版本在Precision、Recall、F1三项指标上的差异，当前生效版本以绿色高亮行与"当前"标签特殊标注，便于管理员评估版本迭代效果，为下一轮模型更新提供决策依据。
5.性能与安全设计
5.1.性能目标
（1）核心业务接口
范围：病例列表查询、病例详情查看、报告保存草稿、报告签发等高频接口。
指标要求：
平均响应时间（RT） < 200ms（不含外部 AI 推理耗时）。
P99（99% 请求）响应时间 < 500ms，在教学机房或院内局域网环境下保持交互流畅。
设计措施：
为病例列表、报告列表等接口建立合理索引（如检查时间、报告状态、签发医生等字段）。
对只读查询采用分页与延迟加载策略，避免一次性加载过多记录。
（2）AI 相关接口
范围：相似病例检索、AI 报告生成、CheXbert 评测触发等依赖外部或自建模型服务的接口。
指标要求：
单次 AI 报告生成整体耗时控制在 5～10 秒内；
后端接口本身快速返回任务受理结果，其后的模型推理过程通过前端轮询或 WebSocket 异步刷新。
设计措施：
将模型调用封装为独立模块，对业务层暴露统一异步接口，避免阻塞主线程。
对重复调用的相似病例检索结果和评测结果进行缓存，减少重复推理。
（3）复杂查询与质控统计接口
范围：按时间、病理类别、模型版本统计 CheXbert 指标的质控报表，典型病例列表、使用日志汇总等多表关联查询。
指标要求：
平均响应时间 < 1.5s，P95 响应时间 < 3s。
设计措施：
在 openGauss 中为统计场景设计专门的汇总表或视图，必要时通过定时任务预计算部分统计结果。
对时间区间、病理标签、模型版本等高频过滤字段建立组合索引。
（4）页面加载性能
放射科工作站首页：首屏加载时间（FCP） < 1.0s，在普通有线/院内网络环境下完全可交互时间（TTI） < 2.0s。
影像详情页：先渲染病例信息与空白影像框架，再异步加载影像文件，保证医生在 1 秒内看到基本结构，影像加载进度有明显提示。
5.2.安全设计
（1）细粒度权限控制（RBAC）
采用基于角色的访问控制模型（RBAC），预置“放射科医生”“管理与质控人员”“系统管理员”等角色。
在接口层控制访问权限：
医生角色可访问病例列表、影像浏览、AI 报告生成与签发相关接口。
管理与质控角色可额外访问质控统计、典型病例管理和部分配置接口。
系统管理员拥有用户管理、角色配置、系统参数维护等高权限操作。
在前端对菜单与按钮进行显示控制：
普通医生仅能看到【生成 AI 报告】【保存草稿】【签发报告】等基础按钮；
管理/质控角色可看到【查看评测详情】【配置参数】等功能；
超级管理员才可进行“清理测试数据”等危险操作。
（2）数据脱敏与隐私保护
病例中仅存储患者匿名 ID，不保存姓名、身份证号等直接身份识别信息；
在需要展示患者相关信息的场景中，统一采用脱敏显示策略，例如只显示“患者 ID：P******01”；
如在科研/教学模式下需要查看更详细信息，必须通过具有特殊权限的账号登录，并在操作日志中记录“查看明文”行为，保证可追溯。
（3）输入验证与接口防护
对所有对外暴露的接口统一进行参数校验：
对 ID 类参数（如病例 ID、报告 ID）进行合法性验证，防止越权访问。
对字符串类参数实施长度与字符集限制，防止 SQL 注入和命令注入。
对富文本输入（报告正文等）进行必要的字符转义与过滤，防止 XSS 攻击。
后端采用统一异常处理机制，将内部异常转换为统一格式的错误响应，避免将数据库结构、堆栈信息等敏感细节直接暴露给前端。
（4）传输与部署安全
建议在实际部署时使用 HTTPS 提供前后端访问通道，防止账号密码和病例数据在网络中以明文传输。
将 openGauss 数据库与 MinIO 对象存储部署在内网，仅对后端应用开放访问权限，避免直接暴露存储接口。
通过容器编排和防火墙规则限制 AI 服务、数据库等内部组件的访问来源，降低误配置或攻击面带来的风险。
（5）日志与审计
对登录、退出、病例查看、AI 报告生成、报告签发、参数修改、用户管理等关键操作，统一记录到操作日志表中，包括用户 ID、操作类型、目标对象、时间、来源 IP 与接口路径。
管理与质控人员可以在后台审计界面按时间、用户、操作类型等条件查询日志，为问题排查和责任
